# Project 2 - TreeBuilder Lexical Analyzer

Authors: Parker Riggs and Ayden Herring

---

## Overview

This is the first of a two-part compiler project. The goal of this stage is to perform lexical analysis on programs written in a custom language called TreeBuilder. The lexer reads source code as input, breaks it into meaningful tokens, and prints each token to standard output. In the next stage, Project 3, these tokens will be passed to a parser built with YACC for syntactic analysis.

The lexer is written using Flex and compiled with GCC. It is designed to recognize all of the keywords, literals, variable names, and symbols that the TreeBuilder language uses.

---

## The TreeBuilder Language

TreeBuilder is a small custom language designed to let you define tree structures where every node has a name and a weight and can have any number of children. Once you have built a tree, you can print it to the screen. The language supports three main kinds of statements.

The first is a buildnode statement, which creates a single node. You give the node a name as a string and a weight as an integer. Optionally, you can designate it as a child of another node using the isachildof field. If you leave isachildof out, the node is treated as a root.

The second is a for loop that iterates over a range of integers. You give it a variable name, a low value, and a high value, and it runs the loop body once for each integer in that range. The loop variable is only accessible inside the loop.

The third is a for loop that iterates over a list of string literals separated by commas. It works the same way as the integer range loop, except the variable takes on each string value in the list during each iteration.

The language also supports a print statement that takes the name of a node and prints the entire tree rooted at that node.

Expressions using the plus operator are supported anywhere a value is expected. Adding two integers produces an integer result. Concatenating strings with plus produces a string. If you mix integers and strings in an expression, the whole thing evaluates as a string.

## Token Types

When the lexer runs, it prints one token per line to standard output. Here is what each token looks like and what it represents.

- BUILDNODE - the buildnode keyword that starts a node definition
- NAME - the name keyword inside a buildnode block
- WEIGHT - the weight keyword inside a buildnode block
- ISACHILDOF - the isachildof keyword inside a buildnode block
- FOR - the for keyword that starts a loop
- IN - the in keyword between the loop variable and the range or list
- PRINT - the print keyword
- STRING: followed by the literal - any double-quoted string value found in the source
- INT: followed by the number - any sequence of digits found in the source
- VAR: followed by the name - any variable name found in the source
- All symbols such as braces, brackets, parentheses, semicolons, equals signs, plus signs, colons, and commas are printed as-is, one per line

---

## Variable Name Rules

A valid variable name must contain at least one letter or underscore. It can start with a letter or underscore and contain letters, digits, and underscores. It can also start with one or more digits as long as it includes at least one letter or underscore somewhere after the leading digits. A name made up entirely of digits is treated as an integer, not a variable.

---

## Comments

The lexer silently discards both single-line comments starting with double slashes and multi-line block comments wrapped in slash-star and star-slash. These are stripped during tokenization and produce no output.

## File Structure

- TreeBuilder.l - the main Flex source file containing all token rules and the main function
- lex.yy.c - the C file generated by Flex from TreeBuilder.l, do not edit this by hand
- lexer - the compiled executable produced by GCC
- input.txt - a sample TreeBuilder program used for testing
- Makefile - handles running Flex and compiling the output automatically

---

## How to Build

You need Flex and GCC installed. If you are on Windows, building through WSL is recommended since the Makefile targets a Unix environment.

To build from scratch, run:

    make

This will run Flex on TreeBuilder.l to generate lex.yy.c and then compile that file into an executable called lexer.

To remove the generated files and start clean, run:

    make clean

---

## How to Run

To run the lexer against the included sample input, run:

    make test

This is equivalent to running:

    ./lexer < input.txt

You can substitute any TreeBuilder source file in place of input.txt. The lexer reads from standard input, so you can pipe any file through it.

---

## Example

Given the following TreeBuilder source code as input:

    buildnode{
    name="root";
    weight=10;
    };

The lexer would produce:

    BUILDNODE
    {
    NAME
    =
    STRING: "root"
    ;
    WEIGHT
    =
    INT: 10
    ;
    }
    ;

---

## Notes

The lexer handles Windows-style line endings (CRLF) transparently, so input files created on Windows will tokenize correctly even when the lexer is run inside WSL or a Linux environment. Carriage return characters are treated as whitespace and ignored.

Whitespace in general, including spaces, tabs, and newlines, is silently discarded and never appears in the token output.

Any character or sequence that does not match any known pattern will produce an UNKNOWN token. If you see UNKNOWN in the output it typically means there is an unrecognized character in the source file.
